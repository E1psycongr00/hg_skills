Codex 커스텀 에이전트는 "좋은 역할 이름"보다 "좋은 경계"가 더 중요하다. 각 에이전트는 좁고 의견이 뚜렷해야 하며, 어떤 일을 맡고 어떤 일은 맡지 않는지가 분명해야 한다.

## 목표

- 반복해서 쓸 수 있는 Codex custom agent TOML을 만든다.
- 사용자 요구를 에이전트의 책임, 입력, 출력, 금지 범위로 변환한다.
- 여러 에이전트를 만들더라도 기본 산출물은 `.codex/agents/*.toml`에 머무르게 한다.

## 기본 범위

- 이 스킬의 기본 산출물은 저장 가능한 custom agent TOML 파일이다.
- 프로젝트 전역 런타임 설정인 `.codex/config.toml`의 `[agents]`, `max_threads`, `max_depth`, `job_max_runtime_seconds`는 기본 범위에 넣지 않는다.
- 부모 프롬프트 예시, 일반 시스템 프롬프트, 워크플로 운영 문서는 기본 산출물에 넣지 않는다.
- 브라우저, MCP, 스킬 같은 의존성이 중요하더라도 우선은 TOML 내부 필드와 `developer_instructions` 안에서 해결한다.

사용자가 전역 설정이나 호출 프롬프트를 정말 명시적으로 요구하더라도, 먼저 agent TOML을 완성하고 그것이 본 작업의 핵심인지 분리해서 다뤄라.

## 진행 순서

### 1. 작업의 반복 패턴을 먼저 잡는다

사용자가 원하는 것은 "어떤 파일 포맷" 자체보다 "앞으로 어떤 일을 계속 맡길 것인가"인 경우가 많다. 먼저 아래 항목을 짧게 정리한다.

- 반복될 작업 한 줄 요약
- 성공 기준
- 읽기 전용인지, 실제 수정이 필요한지
- 외부 문서, 브라우저, MCP, 특정 도구가 필요한지
- 실행 전제가 필요한지
- 빠른 응답이 중요한지, 깊은 추론이 중요한지

정보가 조금 비어 있어도 멈추지 말고 합리적인 기본값을 채운 뒤 가정을 명시한다. 다만 샌드박스, 쓰기 권한, 네트워크처럼 위험이나 비용에 영향을 주는 요소는 흐리게 두지 않는다.

### 2. 역할은 업무 종류가 아니라 경계로 나눈다

역할 분해는 "탐색", "검증", "구현", "문서 확인"처럼 책임이 갈라지는 지점을 기준으로 한다.

- 읽기 중심 탐색과 코드 수정은 가능한 한 분리한다.
- 리뷰와 구현을 한 에이전트에 같이 넣지 않는다.
- 문서 조사나 외부 API 확인이 필요하면 별도 조사 에이전트를 둔다.
- 병렬 쓰기 작업은 파일 소유권이 겹치지 않을 때만 허용한다.
- 하나의 에이전트가 탐색, 구현, 검증, 배포까지 다 하도록 만들지 않는다.

여러 에이전트를 제안할 때는 "왜 이 분해가 토큰, 충돌, 품질 면에서 더 낫나"를 한두 문장으로 설명한다.

### 3. 각 에이전트 스펙을 필드 단위로 결정한다

정확한 필드와 기본 동작은 `references/openai-subagents-spec.md`를 기준으로 잡는다.

각 에이전트마다 아래를 결정한다.

- `name`
- `description`
- `developer_instructions`
- 필요 시 `model`
- 필요 시 `model_reasoning_effort`
- 필요 시 `sandbox_mode`
- 필요 시 `nickname_candidates`
- 필요 시 `mcp_servers`
- 필요 시 `skills.config`

필수 규칙:

- `.codex/agents/<basename>.toml`의 `<basename>`과 TOML 내부 `name`은 완전히 같아야 한다. 하이픈과 언더스코어 차이도 허용하지 않는다.
- `name`은 짧고 안정적인 식별자로 잡는다.
- `description`은 "이 에이전트를 언제 꺼내야 하는지"를 짧게 설명한다.
- `developer_instructions`는 "선택된 뒤 어떻게 행동해야 하는지"만 쓴다.
- 탐색용 에이전트는 가능한 한 `read-only`에 둔다.
- 수정 에이전트는 `workspace-write`를 쓰되, 파일 소유 범위를 분명히 적는다.

### 4. 모델과 추론 강도는 역할에 맞춘다

기본 선택은 `references/openai-subagents-spec.md`의 가이드를 따른다.

- 코디네이션, 애매한 판단, 복합 디버깅: `gpt-5.4`
- 읽기 중심 탐색, 대량 스캔, 보조 조사: `gpt-5.4-mini`
- 지연 시간이 매우 중요하고 텍스트 위주 반복 작업: `gpt-5.3-codex-spark`
- 복잡한 리뷰, 보안, 로직 추적: `high`
- 대부분의 일반 작업: `medium`
- 단순하고 빠른 처리: `low`

사용자가 속도와 비용을 더 중시하면 더 가벼운 모델을, 근거와 정확성을 더 중시하면 더 강한 모델과 높은 추론 강도를 고른다.

### 5. `developer_instructions`는 행동 경계가 보이게 쓴다

좋은 `developer_instructions`는 긴 설명보다 선명한 운영 원칙을 준다. 다음 재료를 우선 포함한다.

- 맡은 일 한 줄
- 우선순위
- 써야 하는 도구나 증거 방식
- 하지 말아야 할 인접 작업
- 결과를 부모 에이전트에게 어떻게 넘길지
- 핵심 도구나 서버가 없을 때 어떻게 실패하고 보고할지

브라우저나 MCP가 agent의 핵심 가치에 직접 영향을 주면, 실행 전제와 fallback을 별도 운영 문서로 빼기보다 우선 `developer_instructions` 안에서 드러내는 편이 좋다.

- 브라우저 agent라면 runnable app, preview URL, 로그인 계정, 수집할 증거 같은 전제를 적는다.
- MCP 기반 조사 agent라면 필요한 서버, 검증 방식, 미가용 시 중단 조건을 적는다.
- 핵심 도구가 없으면 추정으로 메우지 말고 차단 요인을 명시하고 부모에게 넘기게 한다.
- 모든 agent에 `mcp_servers`를 기계적으로 붙이지 않는다. 핵심 의존성이 있을 때만 쓴다.

병렬 수정 에이전트라면 다음 성질도 반영한다.

- 다른 에이전트와 같은 코드베이스에서 함께 일한다.
- 자기 소유 범위 밖의 파일이나 변경을 되돌리지 않는다.
- 충돌 가능성이 보이면 수정 대신 요약과 리스크를 남긴다.

### 6. 출력은 TOML 중심으로 정리한다

최종 응답은 `assets/custom-agent-output-template.md` 형식을 따른다. 기본 산출물은 다음이면 충분하다.

- 짧은 설계 요약
- 저장 가능한 TOML 파일 본문
- 남아 있는 가정이나 결정 포인트

여러 파일을 만들 때는 각 파일 경로를 명시한다. 예시는 바로 저장 가능한 상태여야 한다.

기본 응답에 `.codex/config.toml`, `[agents]` 스니펫, 부모 프롬프트 예시를 넣지 마라. 사용자가 그것도 분명히 원할 때만 "추가 산출물"로 분리해서 덧붙인다.

## 작성 원칙

- 범용 만능 에이전트보다 좁고 선명한 에이전트를 우선한다.
- 사용자 작업을 기준으로 설명하고, 포맷 설명만 길게 늘어놓지 않는다.
- 읽기 전용 작업을 기본값으로 생각하고, 쓰기 권한은 이유가 있을 때만 올린다.
- 병렬화 이점이 작으면 에이전트를 늘리지 않는다.
- 프로젝트 전역 설정을 기본 결과에 끌어들이지 않는다.
- 출력 계약이 흐릿하면 나중에 부모 에이전트가 다시 정리하느라 비용이 커진다.

## 자가 점검

- 각 TOML에 `name`, `description`, `developer_instructions`가 모두 들어갔는가
- 각 TOML 파일의 basename과 내부 `name`이 완전히 같은가
- `description`은 호출 시점 기준이고, `developer_instructions`는 실행 시점 기준인가
- 각 에이전트가 하나의 핵심 책임만 갖는가
- 쓰기 권한이 필요한 에이전트만 `workspace-write`를 쓰는가
- MCP나 스킬은 필요한 에이전트에만 붙였는가
- 브라우저/MCP 의존 agent라면 실행 전제와 fallback이 `developer_instructions`나 TOML 필드 안에서 보이는가
- 병렬 쓰기라면 파일 소유 경계와 충돌 회피 지침이 보이는가
- 기본 응답이 `.codex/agents/*.toml` 생성에 집중되어 있는가

## 참조 파일

- 포맷, 필수 필드, 선택 필드 판단이 필요할 때 `references/openai-subagents-spec.md`
- 최종 응답 모양을 일정하게 맞출 때 `assets/custom-agent-output-template.md`
