새 스킬을 만들고, 테스트하고, 반복적으로 개선할 수 있는 스킬이어야 한다.

## 추구하는 좋은 스킬이란

- 재사용 가능한 절차 지식 패키지
- SKILL.md는 짧아도 방향이 선명해야 한다. 길이를 채우기보다 목적, 품질 기준, 공통 제약, 기본 워크플로를 먼저 드러내라.
- SKILL.md에 남는 문장은 대부분의 실행에서 반복해서 쓰이는 공통 운영 원칙이어야 한다.
- 특정 도메인 디테일, 긴 예시, 변형별 규칙은 `references/`로 보내고, 결정적 실행이나 검증은 `scripts/`로 보낸다.
- 출력 형식의 일관성이 중요하면 `assets/`로 분리한다.
- 점진적 로딩(필요할때 동적 로딩)
- 언제 trigger되고 언제 trigger가 안되야 하는지 분명해야 한다.
- description은 라우팅 메타데이터다. 실행 계획이나 운영 절차를 설명하는 곳이 아니다.
- 적은 설명으로도 목표(목적)에 맞는 절차를 재현할 수 있는 운영 패키지여야 한다.
- 일관적인 품질은 많은 규칙 나열보다 좋은 분리와 점진적 로딩에서 나온다.
- 추상적인 가치 문장보다 모델의 선택을 바꾸는 운영 문장을 우선한다.
- 스킬은 세세한 방법과 규제에 대해 포커스를 두고 정의하는 것이 아니다. 일관된 품질을 얻기 위해 문제를 다루는 법에 대해 정의하는 것이다.

## 하는 일

- 사용자 소통하기
- 스킬 생성하기
- 스킬 개선하기
- 스킬 평가하기

사용자가 스킬 생성, 및 스킬 개선이 명확해 보여도, 검토하고 진행해야한다. 기본적으로 사용자 요구사항이 스킬에 적합한지 의심하고, 항상 생성, 개선, 평가 이전에 소통을 시도해야 한다. 사용자가 강하게 생성,개선, 평가를 요청하더라도, 소통이 우선이다.

---

## 사용자 소통하기

작업을 시작하기 전에 사용자 요구사항으로부터 스킬 작성 또는 재작성에 필요한 맥락을 명확히 얻어야 한다. 좋은 스킬을 만들기 위한 맥락 수집을 분명히 하고, 사용자의 요구사항을 까다로울정도로 비판적으로 읽어야 하며, 아쉬운 점이나 위험한 점도 함께 짚어라.

소통할 때는 전문 용어를 늘어놓기보다, 사용자가 지금 결정을 내리는 데 필요한 정보만 정확하게 전달하는 것이 중요하다. 확신이 없다면 용어를 짧게 풀어서 설명하고, 이해할지 애매하면 간단한 정의를 덧붙여도 좋다.

소통의 신뢰성을 더하기 위해 반드시 웹서치 사전 조사를 진행해야 한다. 도메인과 스킬 작업의 깊은 이해를 동반할때도 많고 까다로울정도 비판적으로 사용자와 소통하기 위해서 필요한 작업이다.

## 스킬 생성하기

### 의도 파악

현재 대화 자체에 이미 스킬로 만들고 싶은 워크플로가 담겨 있을 수도 있다. 사용자가 "이걸 스킬로 만들어줘"라고 말하면, 먼저 대화 기록에서 다음 정보를 최대한 추출해라.

- 어떤 도구를 썼는지
- 어떤 단계 순서로 진행했는지
- 사용자가 어떤 수정을 요구했는지
- 어떤 입력/출력 형식이 관찰됐는지

비어 있는 부분만 사용자에게 물으면 된다. 특히 다음 두 가지는 초기에 분명히 해 둬라.

1. 이 요구가 좋은 스킬 후보인지, 불필요하거나 위험한 부분은 없는지
2. 스킬이 제대로 동작하는지 확인할 테스트 케이스를 만들지

### 인터뷰와 리서치

엣지 케이스, 입력/출력 형식, 예시 파일, 성공 기준, 의존성을 먼저 적극적으로 질문하세요. 이 부분이 정리되기 전에는 테스트 프롬프트를 쓰지 마세요.

기존 스킬 초안이나 유사 스킬이 있다면 먼저 읽고, 필요한 경우 관련 스크립트와 참조 문서도 확인해라. 이미 환경에서 확인할 수 있는 정보는 다시 사용자에게 묻지 않는 편이 좋다.

### SKILL.md 작성하기

사용자 인터뷰 내용을 바탕으로 다음 요소를 채우세요.

- **name**: 스킬 식별자
- **description**: 스킬 선택을 돕는 짧은 라우팅 메타데이터. 무엇을 하고 언제 써야 하는지만 적는다.
- **compatibility**: 필요한 도구나 의존성(선택 사항이며, 실제로는 드물게만 필요하다)
- **나머지 스킬 본문**: 스킬이 이미 선택된 뒤 따라야 하는 실행 지침

#### description과 본문 경계

- `description`은 스킬을 고르는 순간에만 쓰이는 라우팅 메타데이터다. 실행 계획, 운영 규칙, 품질 확보 절차를 싣는 자리가 아니다.
- `description`의 책임: 언제 이 스킬을 써야 하는지, 언제 쓰지 말아야 하는지, 어떤 경계 사례에서 이 스킬이 맞는지 요약한다.
- 기본 형식은 `무엇을 한다 + 언제 쓰는가 + 필요하면 얇은 경계 한 줄` 정도면 충분하다. 보통 1~3개의 짧은 문장으로 끝내라.
- `description`에 넣을 것: 사용자 의도, 핵심 산출물/행동, 대표적인 트리거 표현, 인접 스킬과의 얇은 경계.
- `description`에서 뺄 것: 기본 스코프, 시간 범위, 단계 순서, 우선순위, 어떤 파일이나 첨부를 먼저 읽는지, 어떤 도구를 쓰는지, 검증 방법, fallback, 품질 확보 방법.
- 문장이 `어떻게`, `무슨 순서로`, `기본적으로 무엇부터`, `최근 N일`, `첨부까지 같이 읽고`에 답하고 있으면 description이 아니라 본문 후보로 봐라.
- `SKILL.md` 본문의 책임: 스킬이 이미 선택된 뒤 무엇을 해야 하는지 설명한다. 작업 순서, 입력/출력 형식, 체크리스트, 참조 파일/스크립트 사용 조건, 방향성등 같은 스킬 운영에 필요한 내용만 둔다.
- 본문에는 `언제 사용`, `사용 범위`, `트리거 조건`, `should trigger`, `when to use`, `do not use` 같은 섹션이나 문장을 만들지 마라. 이런 내용은 description에서 작성하는 것이다.
- 본문 최종 점검 때는 "이 문장이 스킬을 고르는 기준인가, 고른 뒤 실행하는 기준인가?"를 확인해라. 고르는 기준이면 description으로 이동한다.

#### description 압축 규칙

- 초안에서 운영 세부를 다 쓴 뒤 그대로 description에 넣지 마라. 먼저 본문으로 보낼 내용을 분리하고, description에는 사용자 의도와 작업 종류만 남겨라.
- description은 "사용자가 왜 이 스킬을 찾는가?"에 답해야지 "스킬이 내부적으로 어떻게 움직이는가?"에 답하면 안 된다.
- 실패한 트리거 케이스를 보고 description을 고칠 때도, 실패 프롬프트의 문구를 그대로 덧붙이지 말고 더 상위의 의도와 범주로 일반화해라.
- description 하나만 읽고도 스킬의 내부 절차가 머릿속에 재생된다면 너무 자세할 가능성이 높다.

나쁜 예:

```text
기본 스코프는 오늘 기준 최근 3일 이내 raw 문서이며, 그 문서에 직접 포함되거나 연결된 이미지와 첨부도 같은 범위 안에서 함께 읽는다. graphify와 저장소 규칙을 먼저 확인한 뒤 AI가 우선 신뢰할 고품질 지식 노트를 만든다.
```

좋은 예:

```text
문서와 관련 자료를 바탕으로 고품질 지식 노트를 만든다. 리서치 정리, 문서 통합, 지식 베이스 초안 작성 요청에 적합하다.
```

최소 압축 예:

```text
고품질 지식 노트를 만든다.
```

#### 내용 배치 우선 원칙

새 내용을 추가할 때는 먼저 `SKILL.md`, `references/`, `scripts/`, `assets/` 중 어디에 두는지가 결정되어야 한다. 문장을 쓰기 전에 배치부터 정하라.

1. 대부분의 실행에서 반복해서 적용되는 목적, 품질 기준, 공통 제약, 기본 워크플로, fallback, 최종 점검인가? 그렇다면 `SKILL.md`에 둔다.
2. 특정 도메인, 프레임워크, 예외 상황, 긴 예시, 변형별 규칙처럼 일부 상황에서만 필요한가? 그렇다면 `references/`로 보낸다.
3. 사람이 길게 설명할수록 흔들리는 절차, 반복 검증, 형식 변환, 결정적인 도구 실행인가? 그렇다면 `scripts/`로 자동화한다.
4. 출력 형태나 응답 골격의 일관성이 핵심인가? 그렇다면 `assets/`의 템플릿으로 분리한다.
5. 어느 쪽에도 선명하게 속하지 않고 추상적인 슬로건만 늘어난다면, 본문에 추가하지 말고 더 구체적인 운영 원칙으로 다시 써라.

`SKILL.md`는 "운영 헌법", `references/`는 "깊이 있는 선택형 문서", `scripts/`는 "흔들리면 안 되는 실행", `assets/`는 "반복되는 출력 형식"이라고 생각하면 판단하기 쉽다.

### 스킬 작성 가이드

#### 스킬의 구성

```text
skill-name/
├── SKILL.md (필수)
│   ├── YAML frontmatter (name, description 필수)
│   └── Markdown 지침 (공통 도메인)
└── Bundled Resources (선택)
    ├── scripts/    - 결정적이거나 반복적인 작업을 처리하는 실행 코드
    ├── references/ - 필요할 때만 읽어들이는 문서(특정 도메인, 깊이 있는 디테일)
    └── assets/     - 출력에 사용하는 파일(템플릿, 아이콘, 글꼴 등)
```

#### 점진적 공개

스킬은 3단계 로딩 구조를 사용한다.

1. **메타데이터** (name + description) - 항상 컨텍스트에 들어간다
2. **SKILL.md 본문** - 스킬이 트리거되면 컨텍스트에 들어간다
3. **번들 리소스** - 필요할 때만 사용한다

##### SKILL.md

스킬의 본문을 담당하며 권장 항목은 다음과 같다.

- SKILL.md는 가능하면 400줄 이하로 유지한다.
- SKILL.md는 간결해야 하지만 비어 있으면 안 된다. 대부분의 실행에서 공통으로 쓰이는 운영 원칙만 남겨라.
- 참조 파일은 언제 읽어야 하는지까지 포함해서 분명하게 가리킨다.
- 큰 참조 파일(300줄 이상)에는 목차를 둔다.
- 무조건 따라야 하는 규칙을 늘어놓지마라.
	- 왜 중요한지 모델이 이해하도록 설명해라. 
	- 스킬을 특정 예시에 과적합시키지 말고 일반적으로 유용하도록 만들어야 한다.
	- 초안을 쓴 뒤에는 한 걸음 물러나 새 눈으로 다시 보고 개선해라.

##### 본문에 보통 드러나야 하는 정보 역할

아래 항목은 `SKILL.md` 본문 기준으로 기본적으로 고려하되, 스킬에 기여하지 않으면 합치거나 생략할 수 있다. 헤더 이름과 순서를 고정하지 말고, 정보 역할이 충족되는지를 먼저 본다.

- 목적 또는 품질 기준: 이 스킬이 어떤 결과를 안정적으로 만들려는지 드러나야 한다.
- 공통 제약: 대부분의 실행에서 항상 적용되는 제한이나 금지 범위를 남긴다.
- 기본 워크플로: 처음부터 끝까지 보편적으로 따르는 흐름이 보여야 한다.
- 분기 또는 참조 기준: 언제 `references/`나 다른 리소스를 읽어야 하는지 보이면 좋다.
- fallback: 핵심 도구나 정보가 없을 때 어떻게 멈추고, 어떻게 보고할지 드러나야 한다.
- 최종 점검: 작업을 끝내기 전에 무엇을 확인해야 하는지 남긴다.

짧은 스킬은 이 역할들을 몇 개의 짧은 문단으로 합쳐도 된다. 중요한 것은 섹션 수가 아니라, 기본 실행 품질에 필요한 정보가 빠지지 않는 것이다. `references/`는 이 본문을 대체하는 문서가 아니라, 특정 상황에서 읽는 확장 문서다.

##### 철학을 운영 문장으로 바꾸기

`SKILL.md`에 들어가는 철학,비전, 추구하는 방향과 같은 문장은 분위기 설명이 아니라 모델이 실제 판단 가이드 및 기준이 되어야 한다. 읽고 나서 모델의 선택이 달라지지 않는 문장은 과감히 줄이거나 지워라.

- "정확하고 친절하게" 같은 추상 문장만 두지 말고, 어떤 상황에서 무엇을 우선하는지 드러내라.
- 좋은 철학 문장은 실패했을 때 무엇이 어긋났는지 설명할 수 있어야 한다.
- 거의 모든 실행에 적용되지 않는 문장이라면 `references/`로 보내거나 삭제를 검토한다.
- 문장을 쓴 뒤에는 "이 문장이 실제로 도구 선택, 분기, 보고 방식, 제약 판단을 바꾸는가?"를 점검한다.

예시:

- 나쁨: "사용자에게 진정성 있게 응답하라."
- 좋음: "근거가 약한 내용은 추정으로 메우지 말고 불확실성을 짧게 밝힌다."
- 나쁨: "높은 품질을 유지하라."
- 좋음: "기본 흐름으로 해결되지 않는 예외는 `references/`를 읽기 전까지 본문에 덧붙이지 않는다."

##### references

하나의 스킬이 일반적이기보다 특정 도메인이나 프레임워크에 대해 디테일한 지침이 필요하다면, 변형별로 나누어 정리하는 편이 좋다.

```text
cloud-deploy/
├── SKILL.md (워크플로 + 선택 기준)
└── references/
    ├── aws.md
    ├── gcp.md
    └── azure.md
```

##### scripts

출력 포맷이나, 변화 없이 명확한 것에 대해 검증하거나, 고정된 도구 사용이 필요하다면, skill 본문에 작성하는것보다 script로 자동화하면 일관적인 대답을 얻을 수 있다. 코드를 통해 일관성이 확보되는 작업이라면 scripts를 활용해야 한다.

##### 출력 템플릿 분리

일관된 명확한 출력 형태가 중요하다면 assets/에 {}-templates.md로 만든다.


#### 놀라움이 없어야 한다는 원칙

스킬에는 악성코드나 익스플로잇 코드, 또는 시스템 보안을 해칠 수 있는 내용이 들어가면 안 된다. 스킬 내용은 설명된 사용자 의도 기준으로 봤을 때 놀랍거나 숨겨진 동작을 해서는 안 된다. 사용자를 오도하는 스킬, 무단 접근이나 데이터 유출 등 악의적 목적을 돕는 요청에는 동조하지 마라.

### 테스트 케이스

스킬 초안을 쓴 뒤에는 실제 사용자가 정말 이렇게 말할 법한 현실적인 테스트 프롬프트를 만든다.

본문, description trigger, references, 출력 일관성 포맷 이렇게 4개에 대해서 고민해 볼 수 있다. 기본적으로 json 형태로 보관하며, 본문의 경우 `common-evals.json`,`description-evals.json`, `special-domain-evals.json`, `output-format-evals.json` 이렇게 4개를 `/evals`에 작성한다.

#### 본문 테스트 케이스

본문의 일반적인 스킬 도메인으로 references를 호출하지 않으면서 기본적인 핵심 도메인에 대해 고민해보고, 무엇을 검증할지에 대해 고민하고 제약사항을 뽑아낸다. 본문 검증이 가능한 상황에 대한 케이스에 대해 깊이 고민해보고 어떻게 제약사항을 둘지 고민해보고 기대 사항들을 작성해본다. 출력 형식보다는 스킬 핵심 도메인 품질 자체에 초점을 맞춘다.

```json
{
  "skill_name": "example-skill",
  "evals": [
    {
      "id": 1,
      "prompt": "사용자 작업 프롬프트",
      "expected_output": "기대 결과 설명",
      "expectations": [
        "- 기대 사항 1",
        "- 기대 사항 2",
        ...
      ]
    }
  ]
}
```

#### description 트리커 테스트 케이스

description이 의도된대로 trigger 되는지 테스트할 항목들이다. 참 4개, 거짓 4개 로 6개의 테스트 케이스를 기본으로 구성한다.

```json
{
  "skill_name": "example-skill",
  "evals": [
    {
      "id": 1,
      "prompt": "사용자 작업 프롬프트",
      "should_trigger": true,
    },
    ...
  ]
}
```

#### references에 등록된 스킬 문서 테스트 케이스

일반적인 테스트 케이스가 아닌 해당 문서에 특화된 테스트 케이스를 고민해보고 생성한다. 이 때 json 내부 형식은 `common-evals.json`과 동일하다.

#### 출력 템플릿 테스트 케이스

`coomon-evals.json`과 동일한 구조를 지니며 출력 검증에 특화된 테스트 케이스를 생성한다.

## 스킬 개선하기

스킬을 개선하는것은 보수적이여야 한다. 요구사항대로 매번 반영하기만 하면 과적합 위험이 있고, 스킬이 복잡해지며, 정체성을 잃을 수 있기 때문이다.

### placement-first 개선 절차

개선 요청이 들어오면 곧바로 `SKILL.md`에 문장을 덧붙이지 말고, 먼저 이번 변경의 자리를 판정한다.

1. 이 변경이 대부분의 실행에서 반복되는 기본 운영 규칙인가? 그렇다면 `SKILL.md` 후보다.
2. 특정 도메인, 예외 상황, 긴 사례 설명인가? 그렇다면 `references/` 후보다.
3. 사람이 설명할수록 흔들리는 반복 절차나 검증인가? 그렇다면 `scripts/` 후보다.
4. 출력 골격이나 문서 포맷의 재사용 문제인가? 그렇다면 `assets/` 후보다.
5. 여전히 배치가 모호하다면, 본문 추가보다 짧은 운영 원칙으로 다시 쓰거나 아예 반영하지 않는 쪽을 우선 검토한다.

본문에 넣기로 결정했다면, 기존 문장을 대체하거나 압축해서 넣는 편을 먼저 시도하라. 새로운 섹션을 늘리는 것은 마지막 선택지다.

### 사용자와 함께 개선

사용자가 요구하는 스킬의 개선은 경험에서 나오는 개선일 경우가 많다. 그래서 대부분의 경우는 불필요한 정보나, 대화 스타일, 또는 퀄리티를 높이기 위해 어떤 행동이나 태도를 보여야 할지, 제약사항, 출력 포맷 변경등, 스킬 시스템에 강력한 변경을 주는 경우가 많기 때문에 고민해야 한다.

- 사용자의 요구를 그대로 본문에 누적하지 말고, 먼저 어떤 종류의 개선인지 분류하라.
- `SKILL.md`가 길어지는 방향보다 더 분명한 분리와 압축으로 해결할 수 없는지 먼저 본다.
- 철학이나 가치처럼 보이는 요청은 실제 운영 문장으로 번역되지 않으면 반영을 보류한다.
- 개선 사항이 오히려 간결함을 해치고 복잡해지지 않는지?
- 정책,방향 느낌이 아니라 너무 고정된 과도한 규칙으로 인해 과적합 문제 또는 오히려 모델의 창의성을 저하시키지 않는지?
- 스킬의 방향성을 해치지 않는지?
- 실패 대응력이 있는가?

고려 사항과 함꼐 트레이드 오프에 대해 고민해보고 사용자와 대화하며 적극적인 개선을 해야 한다.

### 스스로 개선

스스로 개선하는 경우 대부분 도메인보다는 최적화의 경우가 많다. 벤치마크나 검증기에게 검증 이후 피드백을 받는 경우인데 다음과 같은 경우에 스스로 개선할 수 있다.

- 본문에 들어간 내용을 `references/`, `scripts/`, `assets/`로 재배치하면 더 간결해질 수 있는 경우
- 추상적인 철학 문장을 더 짧고 선명한 운영 문장으로 바꿀 수 있는 경우
- 반복되는 실수
- 같은 내용을 더 간단하게 스킬 작성 가능
- 더 적은 토큰으로 비슷한 품질을 낼 수 있음
- 많은 입력때문에 발생하는 문제를 사용자의 적은 프롬프트 입력으로도 이해하고 적용 가능하게 할 수 있는 경우
- 검증기, 또는 벤치 성능 이슈로 개선해야 하는 경우

## 테스트 케이스 실행 및 검증

### 검증 산출물 원칙

검증을 위해 파일, 폴더, 문서, 이미지 같은 결과물을 만들어야 한다면 기본값은 `임시 생성 -> 확인 -> 삭제 -> 피드백`이다. 검증 과정에서 만든 산출물은 평가가 끝난 뒤 작업 디렉터리에 남아 있으면 안 된다.

- 검증용 산출물은 기존 사용자 파일과 섞지 말고, 임시 디렉터리나 명확한 scratch 경로에서만 생성한다.
- 서브 에이전트에 테스트를 맡길 때도 생성 가능한 산출물 경로와 삭제 책임을 프롬프트에 명시한다.
- 결과를 확인한 뒤에는 피드백에 필요한 정보만 남기고, 생성한 산출물은 바로 삭제한다.
- 사용자가 명시적으로 보존을 요청한 경우에만 검증 산출물을 남길 수 있다.
- 삭제가 실패하거나 정리되지 않은 파일이 남으면, 남은 경로와 이유를 사용자에게 즉시 보고한다.

### skill 검증

테스트 케이스를 실행하는 방법은 서브 에이전트를 스폰해서 각 테스트 케이스 하나씩 돌린다. 돌리기 전에 각 케이스가 어떤 임시 산출물을 만들 수 있는지 예상하고, 생성 위치와 삭제 계획을 먼저 정한다. 돌리고 나면 어떤 결과가 나왔고, 토큰은 얼마나 사용하고 시간은 얼마나 사용했는지 기록하는 방식으로 한다. 만약 스킬 사용 버전과 스킬 미사용 버전의 테스트 케이스를 비교하고 싶다면, 해당 skill을 사용하지 않는다고 명시하고 서브 에이전트로 테스트를 진행한다.

등록된 테스트 케이스의 expectations가 얼마나 일치하는지 서브 에이전트를 통해 검증하고, 결과를 요약해서 보여준다. 이때 요약에는 생성한 임시 산출물, 삭제 여부, 삭제 실패가 있었는지까지 함께 포함한다.

### description 검증

description은 프롬프트를 입력할때 마지막에 실제 스킬 행동을 하게 하지 말고, 트리거만 확인되면 종료될 수 있도록 프롬프트에 일부를 첨가해서 서브에이전트에 넘겨서 검증해본다. 토큰 소비와 시간 소비를 줄이려는 목적이다. 각 케이스별로 에이전트를 스폰해야 한다. description 검증에서도 부수적으로 만든 파일이나 로그가 있다면 바로 삭제하고, 끝나면 에이전트를 종료한다.

### 빠른 구조 검증기

`quick_validate.py` 같은 빠른 구조 검증기는 형식과 구조만 본다. 예를 들면 frontmatter 존재 여부, `name`과 디렉터리 이름 일치, `description`과 `compatibility` 길이 제한, 본문 존재 여부, 과하게 긴 본문, trigger 전용 heading 같은 구조 항목만 검사한다.

이 검증기는 "이 단어가 들어갔으니 잘못" 같은 방식으로 의미를 판정하지 않는다. 문장 자체의 좋고 나쁨, 문맥상 description다운지, 본문이 운영 지침으로 충분한지는 Codex 기반 검증기로 넘긴다.

### 빠른 Codex 검증기

문맥 판단은 session 없는 `codex exec --ephemeral` 기반 Python 검증기로 분리한다. 기본값은 `description` 검증기이며, 각 검증기는 짧은 verdict와 핵심 이슈 1~3개만 반환하게 설계한다.

## 검증기 검증

이미 등록된 검증기를 사용할 수 있고, 사용자가 명시적으로 이런 검증기 에이전트를 사용하거나, 이런 형태의 검증이 필요하다고 말하면, 해당 에이전트를 스폰해서 검증한다. 이때도 검증기가 만드는 결과물은 임시 산출물로 취급하고, 확인이 끝나면 삭제 여부까지 보고받는다. 검증 이후 에이전트를 닫는다.
